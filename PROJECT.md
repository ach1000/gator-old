## Project Overview

**Gator** is a CLI RSS feed aggregator written in Go. It fetches RSS feeds from URLs, parses posts, and stores them in a PostgreSQL database. The program runs long-lived aggregation loops that continuously fetch feeds on a configurable schedule.

### Tech Stack
- **Language**: Go 1.x
- **Database**: PostgreSQL
- **Migrations**: Goose
- **Database Code Generation**: sqlc

## Core Components

### 1. State Management (`main.go`)

**`state` struct**: Holds application state
```go
type state struct {
    cfg  *config.Config      // Configuration (DB URL, current user)
    db   UserStore           // Database interface
    cmds *commands           // Command registry
}
```

**`UserStore` interface**: Abstracts database operations
- All database interactions go through this interface
- Implemented by `*database.Queries` (generated by sqlc)
- Key methods include: `GetUser()`, `CreateFeed()`, `GetNextFeedToFetch()`, `MarkFeedFetched()`, etc.

### 2. RSS Feed Aggregation (`rss.go`)

**`fetchFeed(ctx context.Context, feedURL string) (*RSSFeed, error)`**
- Makes HTTP GET request to RSS feed URL
- Sets User-Agent header to "gator"
- Parses XML response into RSSFeed struct
- Unescapes HTML entities in titles and descriptions
- Returns parsed feed or error if fetch/parse fails

**`scrapeFeeds(s *state) error`** (Updated April 2026 to save posts)
- Core aggregation function that fetches feeds and stores posts
- Flow:
  1. Gets next feed to fetch from DB using `GetNextFeedToFetch()`
  2. Fetches the feed via `fetchFeed()`
  3. **For each post item**:
     - Prints title to console for monitoring
     - Parses `published_at` time (tries RFC1123Z format, falls back to alternative)
     - Creates database post record via `CreatePost()`
     - Handles duplicate URL constraint gracefully (continues on error)
     - Logs other errors but continues fetching remaining items
  4. Marks feed as fetched via `MarkFeedFetched()` with current timestamp
- Handles errors gracefully (prints but doesn't crash the loop)
- **Date Parsing**: Tries RFC1123Z format first, then alternative format. If both fail, `published_at` is NULL

**`handlerAgg(s *state, cmd command) error`** (Updated April 2026)
- Command handler for `agg` CLI command
- Takes one argument: `time_between_reqs` (duration string like "1m", "10s", "1h")
- Parses duration using `time.ParseDuration()`
- Prints startup message: "Collecting feeds every Xs"
- Runs infinite ticker loop:
  - Calls `scrapeFeeds()` immediately (doesn't wait for first tick)
  - Then calls `scrapeFeeds()` each time ticker fires
  - Catches and prints errors but continues running
  - Can be stopped with Ctrl+C

### 3. Database Schema

**Tables**:
- `users` - User accounts
- `feeds` - RSS feed URLs and metadata
- `feed_follows` - Many-to-many relationship (user follows feed)
- `posts` - Individual RSS post items (added April 2026)

**Key Columns**:
- `feeds.last_fetched_at` (TIMESTAMP, nullable) - When feed was last fetched
  - NULL initially (never fetched)
  - Set to current time when marked as fetched
  - Used to determine feed fetch order
- `posts` table columns:
  - `id` (UUID) - Unique post identifier
  - `created_at`, `updated_at` - Record metadata
  - `title` (TEXT) - Post title
  - `url` (TEXT UNIQUE) - Post URL, must be unique (prevents duplicates)
  - `description` (TEXT, nullable) - Post description/body
  - `published_at` (TIMESTAMP, nullable) - When post was published (may fail to parse)
  - `feed_id` (UUID FK) - Reference to the feed this post came from

### 4. SQL Queries (`sql/queries/feeds.sql`)

**`GetNextFeedToFetch :one`** (Added April 2026)
- Returns the next feed that should be fetched
- Query: `ORDER BY last_fetched_at ASC NULLS FIRST LIMIT 1`
- Feed selection priority:
  1. **Unfetched feeds first**: Any feed with NULL `last_fetched_at` (never fetched)
  2. **Then by age**: Among fetched feeds, oldest `last_fetched_at` first
  3. **Single result**: LIMIT 1 returns only one feed per call
- Returns `database.Feed` struct with all columns including `last_fetched_at`
- Strategy prevents any single feed from starving others in the rotation

**`MarkFeedFetched :exec`** (Added April 2026)
- Takes feed ID and timestamp
- Sets both `last_fetched_at` and `updated_at` to the provided timestamp
- Used after successfully scraping a feed

**`CreatePost :one`** (Added April 2026)
- Inserts a new post into the database
- Takes all post fields: id, created_at, updated_at, title, url, description, published_at, feed_id
- URL must be unique (UNIQUE constraint) - duplicate URLs cause constraint violation
- Handles duplicate detection at application level (caught in scrapeFeeds)
- Returns the created `Post` record

**`GetPostsForUser :many`** (Added April 2026)
- Returns posts from feeds the current user follows
- Query joins posts → feeds → feed_follows to find user's feeds
- Ordered by `published_at DESC` (most recent first)
- Takes `UserID` and `Limit` (int32) parameters
- Returns slice of `Post` records
- Used by browse command to display recent posts

5. **005_posts.sql** (Added April 2026) - Creates `posts` table for storing RSS items

**To apply migrations**:
```bash
goose -dir sql/schema postgres "postgres://user:pass@localhost:5432/dbname" up
```

**After migrations, regenerate sqlc code**:
```bash
sqlc generate
```

## Code Generation Pipeline

1. **Write SQL** in `sql/queries/` files (e.g., `feeds.sql`)
2. **Run sqlc**: `sqlc generate`
3. **Generated code** appears in `internal/database/`
   - `feeds.sql.go` - Generated query functions
   - `models.go` - Generated struct types
   - `users.sql.go` - User-related queries
4. **Update UserStore interface** in `main.go` to match generated signatures
5. **Use in handlers** via `s.db.QueryName(...)`

**Return Types Note**: sqlc generates specific row types for queries:
- `:one` queries return named row types: `CreateFeedRow`, `GetFeedByURLRow`, etc.
- `:many` queries return slices of row types or model types
- `:exec` queries return just an error
- The base models (`Feed`, `User`, `FeedFollow`) are in `models.go`

**Important**: Generated code is auto-created. Do NOT edit it manually. Always edit the `.sql` files and regenerate.

## Key Assumptions & Design Decisions

1. **Single database connection**: One connection pool shared across all operations
2. **Simple feed rotation**: Always fetches the oldest unfetched feed (or any unfetched if available)
   - This prevents one frequently-updated feed from starving others
3. **Stateless design**: No in-memory cache; each `scrapeFeeds()` call queries DB fresh
4. **Error handling**: Aggregation loop catches errors per-feed but continues running
   - Bad RSS URL won't crash the whole program
5. **Console output**: All post titles printed to stdout for visibility
   - Useful for monitoring and preventing DOS (user can Ctrl+C if too fast)
6. **User-Agent header**: All HTTP requests identify as "gator"
   - Respectful scraping; some servers may reject requests without proper user-agent
7. **No feed storage**: Posts are fetched and printed but not stored in database yet
   - Future enhancement: create `posts` table to store items

## How to Test Changes

### Unit Tests
The test file (`main_test.go`) uses a `mockStore` struct that implements the complete `UserStore` interface:
- Stores users, feeds, and follows in memory (maps and slices)
- Implements all database methods including `GetNextFeedToFetch` and `MarkFeedFetched`
- `GetNextFeedToFetch` mock returns unfetched feeds first (NULL `last_fetched_at`), then oldest
- `MarkFeedFetched` mock updates feed timestamps
- Run all tests: `go test -v`

### Integration Testing
1. **Add feeds** (if needed):
   ```bash
   ./gator login testuser
   ./gator addfeed "Feed Name" "https://example.com/feed.xml"
   ```

2. **Test aggregation** with short interval:
   ```bash
   timeout 30s ./gator agg 5s
   # Or press Ctrl+C manually
   ```

3. **Check database state**:
   ```bash
   psql -U postgres -d gator -c "SELECT id, name, url, last_fetched_at FROM feeds;"
   ```

## Browse Command Details (Added April 2026)

**`handlerBrowse(s *state, cmd command, user database.User) error`**
- Displays recent posts from feeds the user follows
- **Parameters**:
  - Optional limit argument (default: 2)
  - Parses string argument to int32 using `strconv.ParseInt()`
- **Flow**:
  1. Parse optional limit parameter (default 2)
  2. Call `GetPostsForUser()` with user ID and limit
  3. For each post, display:
     - Title (emphasized with ===)
     - Feed ID
     - URL
     - Description (if available)
     - Published timestamp (formatted as RFC1123Z)
- **Usage**:
  ```bash
  ./gator browse           # Shows 2 most recent posts
  ./gator browse 5        # Shows 5 most recent posts
  ./gator browse 10       # Shows 10 most recent posts
  ```
